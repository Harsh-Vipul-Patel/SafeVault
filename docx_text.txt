Each module has a clearly defined scope, internal logic, and database interaction pattern.2.1 Core Banking EngineThe Core Banking Engine is the heart of Suraksha Bank, responsible for the actual movement of funds between accounts. All monetary operations are atomic, logged, and irreversible by design.2.1.1 Internal Fund Transfer FlowINTERNAL TRANSFER — Stored Procedure: sp_internal_transfer()  INPUTS: sender_account_id, receiver_account_id, amount, initiated_by_user_id  SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;    |    +-- SELECT balance FROM ACCOUNTS WHERE id = sender_id FOR UPDATE    |      --> Acquires row-level exclusive lock (prevents race conditions)    |    +-- Validate: balance - amount >= minimum_balance_limit    |      --> ROLLBACK + RAISE EXCEPTION if fails    |    +-- SELECT status FROM ACCOUNTS WHERE id = receiver_id FOR UPDATE    |      --> Verify receiver account is ACTIVE    |    +-- UPDATE ACCOUNTS SET balance = balance - amount WHERE id = sender_id    +-- UPDATE ACCOUNTS SET balance = balance + amount WHERE id = receiver_id    |    +-- INSERT INTO TRANSACTIONS (debit record for sender)    +-- INSERT INTO TRANSACTIONS (credit record for receiver)    |    +-- INSERT INTO TRANSFER_LOG (linking both transaction IDs)    |  COMMIT    --> Trigger: trg_audit_balance_change fires AFTER UPDATE on ACCOUNTS2.1.2 Deposit & Withdrawal OperationsDeposit: Application tier validates amount > 0 and account status = ACTIVE. Stored procedure sp_deposit() updates balance, inserts TRANSACTION record with type='CREDIT', fires audit trigger.Withdrawal: Application tier pre-checks balance. Stored procedure sp_withdraw() acquires SELECT FOR UPDATE lock, re-validates balance atomically (prevents TOCTOU race), updates balance, inserts TRANSACTION record with type='DEBIT'.Time-of-Check vs Time-of-Use (TOCTOU): The application-layer balance check is an optimization only. The authoritative check happens inside the stored procedure under a row lock, preventing any race condition between check and update.2.1.3 Interest Accrual Engine — Chunked Batch ModelLeCun Correction: A naive midnight bulk-update of all accounts is a catastrophic design flaw at scale. Suraksha Bank instead uses a chunked batch architecture that spreads write load across the overnight window, preventing I/O spikes and lock contention on the live ACCOUNTS table.Partitioning Strategy: Accounts are divided into 10 logical buckets via MOD(CAST(RIGHT(account_number, 4) AS INTEGER), 10). Each bucket contains approximately 10% of all accounts, giving each job a bounded, manageable workload.Scheduling: Ten Oracle Scheduler jobs (created via DBMS_SCHEDULER.CREATE_JOB) fire hourly from 22:00 to 07:00. Each calls sp_accrue_interest_chunk(p_bucket_id => N), processing only its assigned bucket. No two chunks overlap. Write pressure on ACCOUNTS is spread across 10 hours instead of hammering all rows in one transaction.ACCRUAL_BATCH_CONTROL Table: Each job INSERTs a row with status=IN_PROGRESS before processing. On success it UPDATEs to COMPLETED. A failed mid-run job leaves status=IN_PROGRESS, allowing a watchdog query to detect and safely re-trigger it. The procedure checks this table before writing any INTEREST_ACCRUAL_LOG rows, guaranteeing idempotency — no account is ever double-accrued.Lazy Calculation View (On-Demand Display): A database VIEW v_account_balance_with_accrued computes effective balance as (balance + days_since_last_accrual * daily_rate) on the fly for display purposes. No writes occur. Physical materialisation to INTEREST_ACCRUAL_LOG happens only at month-end via the chunk procedure.Auditability: Every accrual INSERT into INTEREST_ACCRUAL_LOG is permanent and immutable. Each row references its batch_run_id from ACCRUAL_BATCH_CONTROL, making every posted rupee traceable to a specific run, bucket, date, and rate.2.2 Branch Operations ModuleSuraksha Bank operates as a single bank with geographically distributed branches. The branch model manages localized teller operations without compromising central data integrity.2.2.1 Branch Data ArchitectureBRANCHES Table: Each branch is uniquely identified by branch_id and ifsc_code. Advanced Database MechanicsThis section constitutes the core academic contribution of the Suraksha Bank project, demonstrating graduate-level mastery of DBMS concepts.3.1 ACID Compliance — Deep Dive3.1.1 Atomicity & Rollback StrategyAll multi-step financial operations are wrapped in a single database transaction. Partial completion is architecturally impossible.Scenario — Internal Transfer Failure: If the debit UPDATE succeeds but the credit UPDATE fails (e.g., receiver account was just closed by a concurrent manager action), the entire transaction rolls back. The sender's balance is restored. No money is created or destroyed.SAVEPOINT Strategy: For complex multi-leg operations (e.g., batch payroll), named SAVEPOINTs allow partial rollback to a known good state without aborting the entire batch:  BEGIN;    SAVEPOINT before_leg_2;    UPDATE ACCOUNTS SET balance = balance - 5000 WHERE account_id = 'ACC-001';    -- Leg 2 fails:    ROLLBACK TO SAVEPOINT before_leg_2;    -- Re-attempt with corrected data or raise exception  COMMIT;3.1.2 ConsistencyCHECK Constraints (LeCun's Enhancements): Database-level constraints that can never be bypassed by application code:ALTER TABLE ACCOUNTS ADD CONSTRAINT chk_balance_non_negative  CHECK (balance >= 0);ALTER TABLE TRANSACTIONS ADD CONSTRAINT chk_amount_positive  CHECK (amount > 0);ALTER TABLE ACCOUNTS ADD CONSTRAINT chk_account_status  CHECK (status IN ('ACTIVE', 'DORMANT', 'FROZEN', 'CLOSED'));ALTER TABLE PENDING_EXTERNAL_TRANSFERS ADD CONSTRAINT chk_ifsc_format  CHECK (REGEXP_LIKE(destination_ifsc, '^[A-Z]{4}0[A-Z0-9]{6}$'));ALTER TABLE EMPLOYEES ADD CONSTRAINT chk_employee_role  CHECK (role IN ('TELLER', 'BRANCH_MANAGER', 'SYSTEM_ADMIN'));ALTER TABLE TRANSACTIONS ADD CONSTRAINT chk_transaction_type  CHECK (transaction_type IN ('CREDIT', 'DEBIT', 'TRANSFER_DEBIT', 'TRANSFER_CREDIT',                              'INTEREST_CREDIT', 'FEE_DEBIT', 'EXTERNAL_DEBIT'));3.1.3 IsolationIsolation Level: SERIALIZABLE is set for all financial write transactions via SET TRANSACTION ISOLATION LEVEL SERIALIZABLE. Oracle implements this using a multiversion read-consistency model (SCN-based), preventing phantom reads and non-repeatable reads without requiring shared locks on read data.Read Operations: READ COMMITTED is used for report generation and balance inquiries, balancing accuracy with performance.-- Setting isolation level per transaction in sp_internal_transfer():SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;  SELECT balance FROM ACCOUNTS WHERE account_id = $1 FOR UPDATE;  -- FOR UPDATE acquires TX-level row lock (Oracle default behaviour)  -- Concurrent transactions attempting same row WAIT until lock releases  ...COMMIT;3.1.4 DurabilityRedo Log (Oracle): Oracle's Redo Log mechanism ensures every committed transaction is written to the redo log before acknowledgment. Even a server crash after COMMIT cannot lose the transaction — the instance recovery process replays redo logs automatically on restart.Backup Strategy: Oracle Data Pump (expdp/impdp) for logical backups; RMAN (Recovery Manager) for physical backups. Point-in-time recovery enabled via Oracle Archivelog mode and RMAN incremental backup strategy.3.2 Concurrency Control — Race Condition PreventionConcurrency is the most dangerous failure mode in a banking system. Suraksha Bank uses a defense-in-depth strategy.3.2.1 The Double-Withdrawal Race ConditionWITHOUT LOCKING (DANGEROUS):  T1 reads balance = ₹1,000                T2 reads balance = ₹1,000  T1 computes new balance = ₹0             T2 computes new balance = ₹0  T1 writes balance = ₹0                   T2 writes balance = ₹0  --> RESULT: ₹2,000 withdrawn, final balance = ₹0 (MONEY CREATED FROM NOTHING)WITH SELECT FOR UPDATE (SURAKSHA APPROACH):  T1: SELECT balance FOR UPDATE (acquires lock, balance = ₹1,000)  T2: SELECT balance FOR UPDATE (BLOCKED — waits for T1)  T1: UPDATE balance = ₹0, COMMIT (releases lock)  T2: SELECT balance FOR UPDATE (lock acquired, reads balance = ₹0)  T2: balance - ₹1,000 = -₹1,000 < 0 --> ROLLBACK + EXCEPTION raised  --> RESULT: Only ₹1,000 withdrawn, T2 correctly rejected. INTEGRITY PRESERVED.3.2.2 Deadlock PreventionConsistent Lock Ordering: All stored procedures that lock multiple accounts always acquire locks in ascending account_id order. This prevents circular wait conditions (the cause of deadlocks).Lock Timeout: NOWAIT or WAIT N clauses are used on SELECT FOR UPDATE. SELECT ... FOR UPDATE NOWAIT raises ORA-00054 immediately if the row is locked; SELECT ... FOR UPDATE WAIT 5 waits up to 5 seconds then raises ORA-30006. The application tier catches these and retries with exponential backoff.Deadlock Detection: Oracle automatically detects deadlock cycles and raises ORA-00060, rolling back the statement (not the entire transaction) of one participant. The application tier catches ORA-00060, rolls back explicitly, and retries with exponential backoff (up to 3 attempts).3.3 Strategic DBMS Features — Triggers, Stored Procedures & Constraints3.3.1 Database Triggers — Where & WhyTrigger NameEventPurposeWhy Trigger (not App Code)trg_audit_balance_changeAFTER UPDATE on ACCOUNTSWrites before/after balance to AUDIT_LOGCannot be bypassed by any application path — fires for ALL updates including direct SQLtrg_audit_status_changeAFTER UPDATE on ACCOUNTS (status column)Logs account freeze/unfreeze events with actor identityEnsures compliance logging even during emergency DBA interventionstrg_prevent_audit_modificationBEFORE UPDATE/DELETE on AUDIT_LOGRaises exception — makes audit log immutableEnforces append-only policy at the database level, immune to application bugstrg_transaction_sequenceBEFORE INSERT on TRANSACTIONSAuto-generates formatted transaction reference TXN-YYYYMMDD-SEQNOEnsures consistent, collision-free reference numbers without application logictrg_account_closure_checkBEFORE UPDATE on ACCOUNTS (status='CLOSED')Rejects closure if balance > 0 or pending transactions existLast line of defense — enforces business rule even if app layer is compromised3.3.2 Stored Procedures — Where & WhyProcedure NameInvoked ByWhat It DoesWhy Stored Proceduresp_internal_transfer()Application APIExecutes full atomic 2-account transfer with locking, validation, and loggingKeeps all transactional logic in one atomic unit; External Transfer Simulation (RTGS/NEFT)Suraksha Bank does not connect to a real interbank network. Instead, the external transfer feature is architecturally simulated using a two-phase state machine pattern within a single database — incorporating Karpathy's recommendation.4.1 Architectural LogicEXTERNAL TRANSFER — Two-Phase State MachinePHASE 1: INITIATION (Immediate — Atomic)  sp_initiate_external_transfer(      source_account_id, amount, destination_ifsc, destination_account_no, purpose)  SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;    --> SELECT balance FROM ACCOUNTS FOR UPDATE    --> Validate: balance - amount >= minimum_balance    --> Validate: destination_ifsc format via REGEXP_LIKE()    --> UPDATE ACCOUNTS SET balance = balance - amount  (funds ESCROWED)    --> INSERT INTO TRANSACTIONS (type='EXTERNAL_DEBIT', status='PENDING')    --> INSERT INTO PENDING_EXTERNAL_TRANSFERS:          transfer_id, source_account_id, amount, destination_ifsc,          destination_account_no, status='PENDING', initiated_at, initiated_by  COMMIT  --> Funds are deducted. Customer cannot double-spend. Transfer is queued.PHASE 2: SETTLEMENT (Manual Simulation or Scheduled Job)  sp_approve_external_transfer(transfer_id, approved_by_manager_id)  BEGIN TRANSACTION    --> SELECT * FROM PENDING_EXTERNAL_TRANSFERS WHERE id = transfer_id          AND status = 'PENDING' FOR UPDATE    --> UPDATE PENDING_EXTERNAL_TRANSFERS SET          status = 'SETTLED',          settled_at = NOW(),          settlement_reference = 'RTGS' || LPAD(NEXTVAL('rtgs_seq')::text, 16, '0')    --> UPDATE TRANSACTIONS SET status = 'COMPLETED' WHERE linked_transfer_id = transfer_id  COMMIT  --> Audit trigger fires. Transfer marked complete with simulated reference.PHASE 2B: REJECTION / REFUND  sp_reject_external_transfer(transfer_id, rejection_reason)    --> UPDATE status = 'REJECTED'    --> UPDATE ACCOUNTS SET balance = balance + amount  (REFUND)    --> INSERT into TRANSACTIONS (type='CREDIT', purpose='EXTERNAL_TRANSFER_REVERSAL')4.2 PENDING_EXTERNAL_TRANSFERS Table DesignTABLE: PENDING_EXTERNAL_TRANSFERS  transfer_id           RAW(16) DEFAULT SYS_GUID() PRIMARY KEY  source_account_id     VARCHAR(20) NOT NULL REFERENCES ACCOUNTS(account_id)  amount                DECIMAL(15,2) NOT NULL CHECK (amount > 0)  destination_ifsc      VARCHAR(11) NOT NULL    CHECK (REGEXP_LIKE(destination_ifsc, '^[A-Z]{4}0[A-Z0-9]{6}$'))  destination_account   VARCHAR(20) NOT NULL  destination_name      VARCHAR(100)         -- Beneficiary name (KYC record)  purpose               VARCHAR(200)  status                VARCHAR(15) NOT NULL DEFAULT 'PENDING'    CHECK (status IN ('PENDING', 'SETTLED', 'REJECTED', 'CANCELLED'))  transfer_mode         VARCHAR(10) NOT NULL CHECK (transfer_mode IN ('RTGS', 'NEFT', 'IMPS'))  initiated_at          TIMESTAMP WITH TIME ZONE DEFAULT SYSTIMESTAMP NOT NULL  initiated_by          VARCHAR(50) NOT NULL  settled_at            TIMESTAMP WITH TIME ZONE  settlement_reference  VARCHAR(30)          -- Simulated UTR number  rejected_at           TIMESTAMP WITH TIME ZONE  rejection_reason      TEXT  INDEX: (source_account_id, status, initiated_at DESC)  INDEX: (status) WHERE status = 'PENDING'   -- Partial index for queue processing4.3 Why This Simulates Real RTGS/NEFT FaithfullyFund Escrow: Deducting funds immediately in Phase 1 mirrors real RTGS behavior — the RBI debits the sender's bank account at initiation, not settlement.Idempotency: The transfer_id (UUID) prevents double-processing.

Role-Based Page Access Layout

This document defines which pages, features, and Oracle PL/SQL stored procedures are accessible to each role. Access is enforced at three layers: the Application Tier (session token + RBAC middleware), the Oracle DB Tier (stored procedure caller validation via SYS_CONTEXT), and the UI Tier (hidden/locked navigation items).

Denied Actions & Reasons
🚫
Cannot View Other Accounts
Stored procedures reject any account_id not linked to the caller's customer_id. Returns ORA-20001, not empty result.
🚫
Cannot Open New Account
Account creation requires a Teller with branch_id context and KYC verification. Not available in self-service portal.
🚫
Cannot Access Audit Logs
AUDIT_LOG and PROCEDURE_EXECUTION_LOG are restricted to Manager/Admin roles. SELECT privilege not granted to customer DB role.
🚫
Cannot Deposit/Withdraw Cash
Physical cash operations (sp_deposit, sp_withdraw) require a Teller's employee session with verified branch_id. Cannot be initiated online.

A Teller's branch_id is set at login via DBMS_SESSION.SET_CONTEXT('SURAKSHA_CTX','branch_id','BRN-MUM-003'). All stored procedures read this context value and reject operations for accounts or customers outside the teller's assigned branch.

Callable Oracle PL/SQL Procedures
💰
sp_deposit(p_account_id, p_amount, p_teller_id)
Credits account balance, inserts TRANSACTION record (type=CREDIT), fires audit trigger. Validates teller's branch_id matches account's home branch or teller has cross-branch override.
TELLERMANAGER
💸
sp_withdraw(p_account_id, p_amount, p_teller_id)
SELECT FOR UPDATE WAIT 5 acquires row lock, re-validates balance atomically (TOCTOU prevention), debits account, inserts TRANSACTION record. Raises ORA-20002 if balance insufficient after lock.
TELLERMANAGER
🆕
sp_open_account(p_customer_id, p_type_id, p_initial_deposit, p_teller_id)
Creates new ACCOUNTS record, inserts initial deposit TRANSACTION, links to home_branch_id from teller's session context. Generates formatted account_number via sequence.
TELLERMANAGER
⬆️
sp_submit_dual_approval(p_operation_type, p_payload_clob, p_requested_by)
INSERTs high-value operation into DUAL_APPROVAL_QUEUE with status=PENDING. The originating teller cannot approve their own submission — enforced by CHECK constraint on approver_id != requested_by.
TELLERCUSTOMER
🪪
sp_verify_kyc(p_customer_id, p_doc_type, p_doc_number_masked, p_expiry_date, p_teller_id)
INSERTs or UPDATEs KYC_DETAILS record. Sets kyc_status = VERIFIED with verified_by = teller_id and verified_at = SYSTIMESTAMP. If previous record existed with status = EXPIRED, flags it as superseded. Audit trigger fires. Cannot be called for accounts belonging to the teller's own CUSTOMERS record.
TELLER
🏦
sp_open_fd(p_customer_id, p_linked_account_id, p_amount, p_tenure_months, p_rate_type, p_teller_id)
Validates customer KYC is VERIFIED. Debits p_amount from linked savings account via sp_withdraw(). Creates FD_ACCOUNTS record with locked_rate (fetched from SYSTEM_CONFIG), maturity_date = SYSDATE + tenure_months, auto_renewal_flag. Returns fd_account_id. Interest accrual starts via existing batch mechanism.
TELLERMANAGER
🔁
sp_open_rd(p_customer_id, p_linked_account_id, p_monthly_instalment, p_tenure_months, p_teller_id)
Creates RD_ACCOUNTS record and automatically creates a STANDING_INSTRUCTION for monthly debit on the instalment_due_day. Oracle Scheduler picks this up and calls sp_execute_standing_instruction() each month. RD closes automatically when all instalments are collected; corpus credited to linked account.
TELLERMANAGER
📒
sp_issue_cheque_book(p_account_id, p_leaves_count, p_teller_id)
Fetches next cheque range from Oracle sequence CHQ_RANGE_SEQ. Inserts CHEQUE_BOOKS row with start_cheque_number, end_cheque_number, issued_at = SYSTIMESTAMP, status = ACTIVE. Validates account is ACTIVE and KYC is not EXPIRED. Raises ORA-20030 if account has FROZEN status.
TELLER
🛑
sp_record_stop_payment(p_cheque_number, p_account_id, p_reason, p_teller_id)
Validates cheque_number falls within an ACTIVE CHEQUE_BOOKS range for p_account_id. Checks CHEQUES table — raises ORA-20031 if cheque is already CLEARED. Inserts STOP_PAYMENT_INSTRUCTIONS (status = ACTIVE). sp_process_cheque_clearing() queries this table before any clearing attempt.
TELLER
🔧
sp_create_service_request(p_customer_id, p_request_type, p_details_clob, p_raised_by)
INSERTs SERVICE_REQUESTS row with request_type IN ('CHEQUE_BOOK','STOP_PAYMENT','KYC_UPDATE','CARD_BLOCK','ADDRESS_UPDATE','COMPLAINT'), status = RAISED, branch_id from session context. Triggers notification log entry. Customers can call this themselves; Tellers call it on behalf of counter customers.

Exclusive Oracle PL/SQL Procedures
✅
sp_approve_dual_queue(p_queue_id, p_manager_id, p_note)
Validates approver's role = BRANCH_MANAGER via SYS_CONTEXT. CHECK constraint ensures approver != original requester. Executes the queued operation atomically then marks queue record APPROVED.
MANAGER ONLY
🔒
sp_set_account_status(p_account_id, p_new_status, p_manager_id, p_reason)
Updates ACCOUNTS.status to FROZEN/DORMANT/CLOSED. trg_audit_status_change fires automatically, capturing old/new status, actor, and timestamp in AUDIT_LOG. Closure rejected if balance > 0.
MANAGER ONLY
🌐
sp_approve_external_transfer(p_transfer_id, p_manager_id)
Phase 2 of two-phase external transfer. Updates PENDING_EXTERNAL_TRANSFERS status to SETTLED, generates simulated UTR reference via Oracle sequence, updates linked TRANSACTION to COMPLETED.
MANAGER ONLY
↩️
sp_reject_external_transfer(p_transfer_id, p_manager_id, p_reason)
Rejects transfer, reverses the escrowed debit by crediting the source account, inserts CREDIT reversal TRANSACTION, updates transfer status to REJECTED. Full atomic rollback simulation.
MANAGER ONLY
📊
sp_generate_branch_mis(p_branch_id, p_from_date, p_to_date)
Queries three Oracle Views in one call: v_loan_interest_income (SUM of interest_component from LOAN_PAYMENTS), v_deposit_interest_expense (SUM from INTEREST_ACCRUAL_LOG), v_branch_fee_income (SUM of penalty_amount + service_charges from TRANSACTIONS and LOAN_PAYMENTS). Returns NII = income − expense, and total fee income for the period. Branch-scoped by p_branch_id — raises ORA-20040 if caller's session branch_id does not match.
MANAGER ONLY
🏦
sp_process_fd_maturity(p_fd_account_id, p_manager_id)
Validates maturity_date <= SYSDATE. Credits principal + accrued interest to linked savings account via sp_deposit(). If auto_renewal_flag = 'Y': creates a new FD_ACCOUNTS record with same tenure and current interest rate, debits amount from same savings account. If auto_renewal_flag = 'N': sets FD_ACCOUNTS.status = MATURED. Oracle Scheduler calls this daily for due FDs; Manager can call manually for override.
MANAGERSCHEDULER
🔧
sp_resolve_service_request(p_request_id, p_resolution_note, p_resolved_by)
Updates SERVICE_REQUESTS.status = RESOLVED, sets resolution_note (VARCHAR2 stored as CLOB), resolved_at = SYSTIMESTAMP, resolved_by = p_resolved_by. Once RESOLVED, UPDATE and DELETE on that row are blocked by trigger trg_prevent_sr_modification. Branch scope validated against resolved_by session context.

System Admin has Oracle DB privileges for DDL changes (via controlled scripts), SELECT on all tables for monitoring, and EXECUTE on configuration/maintenance procedures. The Oracle DB role SURAKSHA_ADMIN_ROLE explicitly excludes EXECUTE on any financial stored procedure (sp_deposit, sp_withdraw, sp_internal_transfer, etc.).

Hard Architectural Boundary — Explicitly Declared Out of Scope
The Loan Manager role is intentionally constrained. The following functions are architecturally excluded — not just access-denied, but absent from the loan module entirely. This is a deliberate design decision that demonstrates real-world module isolation.
✗ Loan Underwriting / Credit Scoring ✗ Collateral Valuation ✗ Risk Assessment Models ✗ External Credit Bureau Queries ✗ Savings/Current Account Operations ✗ Cash Deposits / Withdrawals ✓ Application Intake & Tracking ✓ EMI Schedule Generation ✓ Loan Disbursement (credit to account) ✓ Repayment Recording (debit from account)
Data Scope: Branch-Scoped Loan Data Only — Zero Access to ACCOUNTS Balance Operations

The Loan Manager's Oracle DB role SURAKSHA_LOAN_ROLE has EXECUTE only on loan stored procedures. Disbursement and repayment call sp_deposit() and sp_withdraw() internally via a definer-rights wrapper — the Loan Manager never directly touches the ACCOUNTS table.

Callable Oracle PL/SQL Procedures
📅
sp_generate_emi_schedule(p_loan_account_id, p_principal, p_annual_rate, p_tenure_months)
Calculates EMI using reducing balance formula: EMI = P × r(1+r)ⁿ / ((1+r)ⁿ−1). Inserts one row per instalment into EMI_SCHEDULE. Wrapped in a single transaction — all rows inserted or none. Called exactly once per loan, before disbursement. Regeneration is permitted only while status = APPROVED (pre-disbursement).
LOAN_MANAGER
💸
sp_disburse_loan(p_loan_app_id, p_loan_mgr_id)
Validates status = APPROVED and EMI_SCHEDULE exists. Amount ≤ SYSTEM_CONFIG('LOAN_AUTO_DISBURSE_LIMIT'): executes immediately via internal call to sp_deposit(). Amount > limit: INSERTs into DUAL_APPROVAL_QUEUE with operation_type = 'LOAN_DISBURSEMENT'. On execution, creates LOAN_ACCOUNTS record, updates application status to DISBURSED, logs disbursement_txn_id.
LOAN_MANAGERMANAGER (approval)
💳
sp_record_emi_payment(p_emi_id, p_loan_mgr_id)
Finds the EMI row, checks due_date vs SYSDATE to compute penalty. Calls sp_withdraw() on the borrower's linked account (definer-rights — Loan Manager never touches ACCOUNTS directly). Updates EMI_SCHEDULE.status = PAID, inserts LOAN_PAYMENTS record, decrements LOAN_ACCOUNTS.outstanding_principal. Full SERIALIZABLE transaction.
LOAN_MANAGER
🔐
sp_close_loan(p_loan_account_id, p_loan_mgr_id)
Verifies ALL EMI_SCHEDULE rows for this loan have status = PAID and outstanding_principal = 0 (enforced by CHECK). Updates LOAN_ACCOUNTS.status = CLOSED. Generates closure certificate reference via Oracle sequence. Audit trigger fires. Raises ORA-20010 if any EMI remains unpaid.
LOAN_MANAGER
🔄
sp_update_loan_status(p_loan_app_id, p_new_status, p_note, p_loan_mgr_id)
State machine enforcement: only valid transitions are allowed (RECEIVED→UNDER_REVIEW, UNDER_REVIEW→APPROVED, etc.). Invalid transitions raise ORA-20011. Each status change INSERTs an immutable record into AUDIT_LOG via trigger trg_audit_loan_status.
LOAN_MANAGER
⏩
sp_loan_prepayment(p_loan_account_id, p_prepayment_amount, p_loan_mgr_id)
Validates p_prepayment_amount < outstanding_principal (full pre-payment must use sp_foreclose_loan). Debits amount from borrower's linked account via internal sp_withdraw() call. Reduces outstanding_principal. Regenerates remaining EMI_SCHEDULE rows with recalculated reducing-balance amounts (using same original rate, new outstanding, remaining tenure). Old schedule rows marked SUPERSEDED. All changes in single SERIALIZABLE transaction — partial failure rolls back entirely.
LOAN_MANAGER
🔚
sp_foreclose_loan(p_loan_account_id, p_loan_mgr_id)
Computes foreclosure amount = outstanding_principal + foreclosure_penalty (rate fetched from FEE_SCHEDULE where fee_type = 'LOAN_FORECLOSURE'). Debits total from borrower's linked account. Marks all remaining PENDING EMI rows as FORECLOSED. Updates LOAN_ACCOUNTS.status = CLOSED with closure_type = FORECLOSED. Generates closure reference via sequence. Audit trigger fires. Raises ORA-20012 if foreclosure penalty computation fails due to missing FEE_SCHEDULE config.
LOAN_MANAGER
🚨
sp_mark_loan_overdue(p_loan_account_id)
Called by Oracle Scheduler daily. Finds all PENDING EMI rows where due_date < SYSDATE and updates status = OVERDUE. Updates LOAN_ACCOUNTS accordingly. If overdue count exceeds SYSTEM_CONFIG('MAX_OVERDUE_EMIS'), sets loan status = DEFAULTED. Inserts into COMPLIANCE_FLAGS table for Branch Manager review.

Denied Actions & Architecture Reasons
🚫
Cannot Touch ACCOUNTS Directly
Disbursement and repayment route through definer-rights wrappers that internally call sp_deposit() and sp_withdraw(). SURAKSHA_LOAN_ROLE has no direct privilege on the ACCOUNTS table.
🚫
Cannot Approve Own Disbursements
High-value disbursements enter DUAL_APPROVAL_QUEUE. The same CHECK constraint already in place (approver_id ≠ requester_id) prevents self-approval. Branch Manager is the approver.
🚫
Cannot Waive EMI Penalty
Penalty amounts are computed deterministically inside sp_record_emi_payment() via date arithmetic. No override parameter exists in the procedure signature for the Loan Manager role.
🚫
Cannot Skip Loan Status Steps
sp_update_loan_status() enforces a strict state machine. Jumping from RECEIVED to DISBURSED raises ORA-20011. Each valid transition is hardcoded in the procedure's CASE statement.
